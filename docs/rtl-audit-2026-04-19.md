# RTL Audit for LibreChat (Botnim deployment) — 2026-04-19

Report produced by a code-only audit subagent. All findings reference the tree at commit 5e537517 on branch `main`. Nothing in this report has been applied.

## Headline findings

**A. The root `<html>` element never gets `dir="rtl"`.** The language selector only writes `document.documentElement.lang` (General.tsx:173, ShareView.tsx:84). There is no code anywhere that sets `document.documentElement.dir` based on language. Because of this, Tailwind's logical utilities (`ms-*`, `me-*`, `text-start`, `text-end`) and the browser's auto-mirroring of `justify-start`/`dir="auto"` blocks never activate for Hebrew users unless they manually toggle the "Chat direction" switch in Settings. Most of what looks like "RTL bugs" in the current deploy are actually "the page is in LTR mode while showing Hebrew content."

**B. There are two independent direction controls that are not connected:**
1. `store.lang` (localStorage key `lang`) — drives `i18next` translations.
2. `store.chatDirection` (localStorage key `chatDirection`, default `'LTR'`) — drives a handful of `isRTL ? … : …` branches *inside* ChatForm / EditMessage / FileRow only. This is a per-user manual toggle buried in Settings → Chat, not auto-derived.

Until (A) or (A+B) is fixed, fiddling with individual Tailwind classes is cosmetic; the big wins come from setting `dir="rtl"` globally.

**C. Hebrew locale exists, but is 280 keys short of English** (`he/translation.json` has 1248 keys vs `en/translation.json` 1528). Several of those keys are on the chat hot path — the message-input aria-label, the agent-handoff banner, stream/refusal errors, and the sidebar open/close tooltip. These will render in English even after the RTL layout is fixed.

## 1. Global direction switch is missing (top priority)

**Files:** `client/src/components/Nav/SettingsTabs/General/General.tsx:165-179` and `client/src/components/Share/ShareView.tsx:73-91`, plus `client/index.html:2`.

`changeLang` and `handleLangChange` both do:
```ts
document.documentElement.lang = userLang;
```
but never touch `dir`. And `client/index.html:2` is hardcoded `<html lang="en-US">`.

**Why it breaks RTL:** Without `dir="rtl"` on the root, every unscoped `ml-*`/`mr-*`/`text-left`/`left-*` in the codebase renders with physical sides, not logical. Browsers also won't flip Unicode bidi embedding or default paragraph direction.

**Proposed fix (one-line inside both callbacks):**
```ts
const rtlLangs = new Set(['he', 'ar', 'fa', 'ug']);
const primary = (userLang.split('-')[0] || userLang).toLowerCase();
document.documentElement.dir = rtlLangs.has(primary) ? 'rtl' : 'ltr';
```
Do this in `changeLang` (General.tsx), `handleLangChange` (ShareView.tsx), and once on boot inside `main.jsx` / `App.jsx` using the initial `store.lang` value so it applies on first paint before the user touches settings. Also change `client/index.html:2` to `<html lang="en-US" dir="ltr">` and let the JS override on boot; alternatively inline a tiny script in `index.html` that reads `localStorage.getItem('lang')` and sets `dir` before React mounts (same pattern as the existing dark-mode FOUC fix on lines 24-49).

**Caveat:** This alone will expose rendering bugs that were hidden by the page being LTR; items 2-9 below should be fixed together or the layout will look worse, not better, on the first build with the global switch enabled.

**Also consider:** Auto-syncing `store.chatDirection` with `store.lang` so that once `lang` becomes `he`, the ChatForm RTL branches (ChatForm.tsx:93-94) activate without the user opening Settings. Cleanest place: inside `useLocalize.ts:14-18` useEffect, or a new tiny hook wired to `App.jsx`.

## 2. Mobile sidebar slides in from the wrong side under RTL

**File:** `client/src/components/Nav/Nav.tsx:260-280`.

```tsx
className={cn('nav fixed left-0 top-0 z-[110] h-full bg-surface-primary-alt', ...)}
style={{
  transform: navVisible ? 'translateX(0)' : `translateX(-${sidebarWidth}px)`,
}}
```

**Why it breaks RTL:** In Hebrew, the sidebar is visually expected on the right. `left-0` + negative X translate slides it off the *left* edge; Hebrew users see a white void on the right and the sidebar appearing from the left. Desktop (line 288-299) has the same problem with `animate={{ x: navVisible ? 0 : -sidebarWidth }}`.

**Proposed fix:** Read `document.documentElement.dir === 'rtl'` (or a `useIsRTL()` hook that subscribes to `store.lang` / `store.chatDirection`) and conditionally use `right-0` + positive X translate for RTL. Suggested sketch:
```tsx
const isRTL = useIsRTL();
// mobile
className={cn('nav fixed top-0 z-[110] h-full ...', isRTL ? 'right-0' : 'left-0')}
style={{ transform: navVisible ? 'translateX(0)' : `translateX(${isRTL ? sidebarWidth : -sidebarWidth}px)` }}
// desktop
animate={{ x: navVisible ? 0 : (isRTL ? sidebarWidth : -sidebarWidth) }}
```

Matching CSS selector in `client/src/mobile.css:23, 40, 152` — `.nav.active { left: 0; }` and mask rules — also need RTL variants (or switch to `inset-inline-start: 0`).

## 3. NavToggle (sidebar open/close button) translates to the wrong side

**File:** `client/src/components/Nav/NavToggle.tsx:49-56`.

```tsx
className={cn(
  className,
  '-translate-y-1/2 transition-transform',
  navVisible ? 'rotate-0' : 'rotate-180',
  navVisible && translateX ? 'translate-x-[260px]' : 'translate-x-0',
)}
```

**Why it breaks RTL:** The button is supposed to sit just outside the sidebar's trailing edge. `translate-x-[260px]` moves it 260px to the right; if the sidebar is now on the right (per fix #2), the toggle appears 260px right of the right edge — i.e. way off-screen. It needs to translate -260px in RTL.

**Proposed fix:** Accept `isRTL` as a prop (passed from Nav.tsx) and render `style={{ transform: \`translateX(${isRTL ? -260 : 260}px) translateY(-50%) rotate(${...})\` }}` instead of a class.

## 5. Conversation title fade-out gradient is on the wrong side

**File:** `client/src/components/Conversations/ConvoLink.tsx:49-57`.

```tsx
<div className={cn(
  'pointer-events-none absolute bottom-0.5 right-0.5 top-0.5 w-20 rounded-r-md bg-gradient-to-l',
  ...
)} />
```

**Why it breaks RTL:** Long titles in the sidebar are truncated with a fade that covers the last 5rem on the *right*. In RTL the truncation happens on the *left*, so the fade sits on the wrong side — you get a faded-out beginning of a title and a hard cut at the end.

**Proposed fix:** Change `right-0.5` → `end-0.5`, `rounded-r-md` → `rounded-e-md`, `bg-gradient-to-l` → keep as `bg-gradient-to-l` in LTR but use a CSS-variable-based gradient direction, or use two classes gated on an `isRTL` prop.

Same issue on the `HoverToggle` at `client/src/components/Conversations/HoverToggle.tsx:10, 26`.

## Recommended ordering

Do these together in one PR; doing any one in isolation may look worse than current state:

1. **Item #1** (global `document.documentElement.dir`) — prerequisite for everything.
2. Auto-sync `chatDirection` with `lang` (trivial wiring in `useLocalize.ts`).
3. **Items #2, #3** (Nav.tsx and NavToggle.tsx) — fixes the dominant layout.
4. **Items #4, #5, #6** (sidebar conversation rows + search bar) — visible on every screen.
5. **Items #13, #14** (CSS `margin-left` → `margin-inline-start`) — cheap, high return.
6. **Item #7** (SourceHovercard logical margins).
7. **Item #10** table (hot-path physical margins → logical).
8. **Items #8, #9** (citation/sibling chevrons).
9. **Item #15** (translation keys).
